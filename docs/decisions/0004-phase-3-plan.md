# ADR 0004 ŌĆö Phase 3 implementation plan (GraphQL codegen package)

- **Status:** Accepted (planning only ŌĆö no code yet)
- **Date:** 2026-05-02
- **Decision owner:** Lead full-stack engineer
- **Supersedes:** Phase plan item 3 of [ADR-0001](./0001-stack.md), refined into
  executable sub-tasks.
- **Depends on:** Phase 2 ([ADR-0002](./0002-phase-2-plan.md) +
  [ADR-0003](./0003-phase-2-amendment.md)) ŌĆö `phase-2-complete` tag must be
  landed before Phase 3 starts.

## Why this plan exists

Phase 3 in [ADR-0001](./0001-stack.md) is one bullet ("GraphQL codegen
package"). In practice Phase 3 is the typed contract that the storefront
(Phase 4ŌĆō6) and any admin-UI extension (Phase 8) consume ŌĆö every shop-api
mutation a customer triggers, every product/collection query an SSR page
renders, and every admin-api call a custom UI extension makes. CLAUDE.md
hard rule #4 mandates that every operation lives in `packages/graphql` and
is codegen'd; this ADR pins the toolchain, fixes the file layout, defines
the operation surface, and decides the schema-snapshot strategy so the
package is reproducible without a running backend.

The package itself (`packages/graphql/`) was scaffolded in Phase 1 as a stub
that re-exports a package marker. Phase 3 fills it in.

## Decision summary

1. **Codegen tool:** `@graphql-codegen/client-preset` (v6) driven by
   `@graphql-codegen/cli` (v7). Output: `TypedDocumentNode` artifacts plus
   the `gql()` tagged-template helper.
2. **Schema source:** committed SDL snapshots at
   `src/schema-snapshots/admin-api.graphql` and `shop-api.graphql`,
   regenerated by `pnpm --filter @hurc/graphql introspect` against a
   running Vendure instance and committed to the repo. `pnpm codegen`
   reads only the snapshots ŌĆö no backend needed for type-generation, lint,
   typecheck, or CI.
3. **Output layout:** two siblings ŌĆö `src/admin/` and `src/shop/` ŌĆö each
   with its own `client-preset` output and its own typed `gql()`. Shared
   scalars live in `src/scalars.ts`. No barrel; consumers import from
   `@hurc/graphql/admin` and `@hurc/graphql/shop` subpaths.
4. **Fragment masking:** **off** for Phase 3 (`useFragmentMasking: false`).
   Storefront uses Server Components that pass plain props down; masking
   adds friction without earning encapsulation. Re-evaluate in Phase 6 if
   we adopt urql's fragment colocation pattern.
5. **Operation surface:** complete shop-api surface needed by Phases 4ŌĆō6
   (browse, cart, checkout, account, GDPR self-service) plus a minimal
   admin-api surface (active administrator, product list/detail) that
   exercises the admin pipeline. Listed in ┬¦3.4.
6. **GraphQL client choice (Phase 4 wiring):** **deferred** to ADR-0005.
   Phase 3 emits `TypedDocumentNode` which is client-agnostic; the choice
   between `urql`, `@apollo/client`, `@tanstack/react-query` + `graphql-request`,
   or a thin RSC fetch wrapper belongs in the storefront ADR, not here.

## Risks called out before code

- **R1 ŌĆö Vendure schema is large.** The combined admin-api + shop-api SDL
  is ~3500 lines and produces ~6000 lines of generated TS. `client-preset`
  output is gzip-friendly (string-keyed maps) but tsserver memory
  noticeably climbs in editors. Mitigation: split into two outputs (`admin/`
  and `shop/`) so the storefront never types-load the admin schema.
- **R2 ŌĆö `@vendure/core` minor bumps shift the schema.** SDL snapshots will
  drift on every Vendure upgrade. Mitigation: a CI check (`pnpm
--filter @hurc/graphql verify-snapshot`) fails the build if the running
  backend's introspection no longer matches the committed snapshot. Land
  in Phase 9 alongside Lighthouse CI.
- **R3 ŌĆö Fragment masking divergence between v5 and v6 of `client-preset`.**
  v6 changed `useFragmentMasking` semantics. We pin v6 explicitly (not a
  caret); document that masking is off (┬¦Decision 4); the v5 ŌåÆ v6 upgrade
  notes are not in scope.
- **R4 ŌĆö Custom HURC scalars.** The HURC plugins do not declare custom
  scalars in Phase 2 (verified: `grep -r 'scalar ' apps/backend/src/plugins`
  returns nothing). Vendure's stock scalars (`DateTime`, `Money`, `JSON`,
  `Upload`) are mapped via `config.scalars` to native TS types. If Phase 4+
  adds a scalar, this ADR must be amended.

`package.json` `exports` map exposes `./admin` and `./shop` subpaths
(plus the existing `.` marker). `files` keeps `dist/` and the
`src/schema-snapshots/*.graphql` (already there from Phase 1) plus
`src/{admin,shop}/operations/*.graphql` so consumers can re-introspect
if needed.

Generated files **are committed**. Reasoning: they're part of the typed
contract; CI without `pnpm codegen` should produce identical output to a
committer's machine. Rationale matches the `dist/` commit policy in
`@vendure/core` itself.

## Sub-task plan

### 3.1 Tooling install + skeleton

1. Add direct dev-deps to `packages/graphql/package.json`:
   - `@graphql-codegen/cli@7.0.0`
   - `@graphql-codegen/client-preset@6.0.0`
   - `@graphql-codegen/schema-ast@6.0.0`
   - `graphql-config@5.1.6`
   - `get-graphql-schema@2.1.2`
   - `graphql@16.13.2` (peer satisfaction; backend already locks this)
2. Replace the placeholder `codegen` script
   (`echo 'codegen wired in Phase 3'`) with
   `graphql-codegen --config codegen.ts`. Add an `introspect` script that
   pipes both endpoints to `src/schema-snapshots/`. Add a
   `verify-snapshot` script that diffs live introspection vs. committed
   snapshot and exits non-zero on drift (used by Phase 9 CI; Phase 3
   ships the script but does not gate CI on it yet).
3. Replace `src/index.ts` to re-export from `./scalars` only (consumers
   pull operations from the subpath exports).

### 3.2 Codegen config (`codegen.ts`)

```ts
// codegen.ts
import type { CodegenConfig } from '@graphql-codegen/cli';

const scalars = {
  DateTime: 'string', // ISO-8601; convert at the edge
  Money: 'number', // integer minor units (CLAUDE.md ┬¦8)
  JSON: 'unknown',
  Upload: 'File',
} as const;

const config: CodegenConfig = {
  overwrite: true,
  config: { scalars, useTypeImports: true, enumsAsTypes: true },
  generates: {
    'src/admin/generated/': {
      schema: 'src/schema-snapshots/admin-api.graphql',
      documents: 'src/admin/operations/**/*.graphql',
      preset: 'client',
      presetConfig: { fragmentMasking: false },
    },
    'src/shop/generated/': {
      schema: 'src/schema-snapshots/shop-api.graphql',
      documents: 'src/shop/operations/**/*.graphql',
      preset: 'client',
      presetConfig: { fragmentMasking: false },
    },
  },
};

export default config;
```

`enumsAsTypes: true` ŌĆö Vendure enums collide with TS-runtime-enum patterns
when imported across package boundaries; string-literal unions sidestep it
and are zero-runtime. Verified against `@vendure/common`'s own approach.

### 3.3 Introspection workflow

```bash
# manual / one-off
pnpm --filter @hurc/backend dev    # in another terminal
pnpm --filter @hurc/graphql introspect
pnpm --filter @hurc/graphql codegen
pnpm --filter @hurc/graphql lint
```

Introspection is a small Node script (`scripts/introspect.mjs`) that posts
`getIntrospectionQuery()` against both endpoints and writes
`printSchema(buildClientSchema(introspection))` to disk. Same code path as
`scripts/verify-snapshot.mjs`, so the two tools cannot disagree on what
"normalized SDL" means ŌĆö a third-party tool like `get-graphql-schema`
would format SDL differently and produce false drift positives.

Vendure's dev server allows unauthenticated introspection on both endpoints
in dev mode, so no auth header is needed. If a future ADR locks
admin-api introspection behind auth, extend `introspect.mjs` to read the
superadmin token from `apps/backend/.env`.

### 3.4 Operation surface

#### Shop-api (storefront, Phases 4ŌĆō6)

- `channel.graphql` ŌĆö `activeChannel`, `availableCountries`
- `customer.graphql` ŌĆö `activeCustomer`, `login`, `logout`, `registerCustomerAccount`, `verifyCustomerAccount`, `requestPasswordReset`, `resetPassword`, `updateCustomer`, `requestUpdateCustomerEmailAddress`, `updateCustomerEmailAddress`
- `address.graphql` ŌĆö `createCustomerAddress`, `updateCustomerAddress`, `deleteCustomerAddress`
- `product.graphql` ŌĆö `productList` (paginated, with `FacetValueFilterInput`), `productBySlug`, `productByIdMinimal`
- `collection.graphql` ŌĆö `collectionList`, `collectionBySlug`, `collectionTreeNav` (for header nav)
- `order.graphql` ŌĆö `activeOrder`, `activeCustomerOrders`, `orderByCode`
- `checkout.graphql` ŌĆö `addItemToOrder`, `adjustOrderLine`, `removeOrderLine`, `removeAllOrderLines`, `applyCouponCode`, `removeCouponCode`, `setCustomerForOrder`, `setOrderShippingAddress`, `setOrderBillingAddress`, `setOrderShippingMethod`, `eligibleShippingMethods`, `eligiblePaymentMethods`, `transitionOrderToState`, `addPaymentToOrder`
- `gdpr.graphql` ŌĆö placeholder query that resolves the customer's own
  data export URL (the actual export is REST per Phase 2 ┬¦2.7). Operation
  exists so the storefront can typecheck the link.

Each operation file exports queries + mutations under named operations
(e.g. `query ProductList(...)`, `mutation AddItemToOrder(...)`). Fragment
discipline:

- One fragment per major entity (`ProductCard`, `ProductDetail`,
  `OrderSummary`, `OrderLine`, `CustomerSummary`, `Address`).
- Fragment names are prefixed by domain when ambiguous (`ShopProductCard`
  vs. an admin counterpart).

#### Admin-api (Phase 3 stub, expanded in Phase 8)

- `administrator.graphql` ŌĆö `activeAdministrator` (proves admin
  auth/codegen pipeline)
- `product.graphql` ŌĆö `adminProductList` (paginated), `adminProductBySlug`
  (used by the Phase 8 GPSR / Meili admin-UI extensions)

The admin-api surface intentionally stays minimal in Phase 3. Vendure's
own Admin UI consumes the admin-api directly; we only need typed
operations for the **custom** admin-UI extensions (GPSR, Meili reindex
button, ResponsiblePerson CRUD) ŌĆö those land in Phase 8 with their own
ADR. Phase 3 ships only enough to prove the admin pipeline compiles.

### 3.5 Build integration (Turbo)

`turbo.json` already declares the `codegen` task with the right outputs
(`src/generated.ts`, `src/**/*.generated.ts`) ŌĆö the second pattern matches
`client-preset` output paths. **Update needed:**

- Change `outputs` from the pre-Phase-3 placeholder shape to
  `["src/admin/generated/**", "src/shop/generated/**"]`. The
  Phase-1-era `src/generated.ts` path is obsolete.
- Add `inputs` so cache invalidates only on real changes:
  `["src/schema-snapshots/**", "src/{admin,shop}/operations/**", "codegen.ts"]`.

`build` and `typecheck` get an explicit `dependsOn: ["codegen"]` so
consumers (`apps/storefront`) trigger codegen automatically. The
`@hurc/backend` does not depend on `@hurc/graphql` and is unaffected.

### 3.6 Lint, format, IDE

- ESLint: extend the package's existing `eslint.config.js` to ignore
  `src/{admin,shop}/generated/**` (codegen output is not human-edited).
- Prettier: `.graphql` files formatted by Prettier's built-in parser
  (`prettier --check 'packages/graphql/src/**/*.graphql'` added to root
  `lint` task).
- VS Code: `graphql.config.ts` (graphql-config v5) wires the GraphQL
  language-server extension so editors get autocomplete on `.graphql`
  files against the committed SDL.

### 3.7 Tests

Unit tests in `packages/graphql/test/`:

- **Snapshot test** (`snapshot.test.ts`) ŌĆö asserts the SDL parses
  cleanly, that `printSchema(buildSchema(...))` is round-trip stable,
  and that HURC custom-field types (`ProductCustomFields`,
  `CustomerCustomFields`, `ResponsiblePerson`) are present on shop-api.
  Catches accidental SDL hand-edits.
- **Operation parse test** (`operations.test.ts`) ŌĆö for each project,
  every `.graphql` file in `src/{admin,shop}/operations/` parses, and
  the **combined document** validates against the committed schema via
  `graphql.validate`. Combined (not per-file) because operations
  cross-reference fragments across files (e.g. `OrderDetailFields` in
  `order.graphql` is used by `checkout.graphql`); per-file validation
  would false-positive on unresolved fragments. The combined check is
  exactly what `client-preset` does at codegen time.
- **Exports test** (`exports.test.ts`) ŌĆö imports
  `@hurc/graphql/{admin,shop}` subpaths, asserts `graphql()` is
  callable, and asserts a known query string returns a defined document.

Coverage gate: 80% of `packages/graphql/src/scalars.ts` (the only
human-written code path that has runtime behavior). Generated files are
excluded from coverage.

## Order of operations (one PR each)

1. `chore(graphql): install codegen toolchain (v7 cli + v6 client-preset)`
2. `feat(graphql): codegen.ts + schema-snapshots scaffolding`
3. `feat(graphql): introspect + verify-snapshot scripts`
4. `feat(graphql): shop-api operation set (channel, customer, product, collection)`
5. `feat(graphql): shop-api operation set (cart, checkout, order, gdpr)`
6. `feat(graphql): admin-api minimal operation set`
7. `feat(graphql): operation parse + snapshot tests`
8. `chore(turbo): wire codegen task graph`
9. `chore(repo): phase-3-complete tag`

Each PR is small, independently revertable, and ends with all repo gates
green.

## Open follow-ups (after Phase 3)

- **ADR-0005 ŌĆö storefront GraphQL client** (Phase 4): pick `urql` vs.
  `@tanstack/react-query` + `graphql-request` vs. a thin RSC fetch
  wrapper. Cache invalidation (`revalidateTag` after mutations), auth
  token attachment, error handling.
- **CI snapshot drift gate** (Phase 9): wire `verify-snapshot` into the
  GH Actions matrix so a Vendure plugin schema change cannot land
  without regenerating SDL.
- **Multi-locale operations** (Phase 4): once `next-intl` lands,
  re-evaluate whether `localeString`/`localeText` selections need a
  `LanguageCode` input parameter on every product query, or whether the
  storefront sets it once via `Channel` + `Vary` cookie. Likely the
  latter.
- **Custom-admin-UI operations** (Phase 8): expand admin-api operations
  for GPSR validation, Meili reindex, ResponsiblePerson CRUD.
- **Persisted operations** (post-launch): if shop-api egress becomes a
  concern, switch `client-preset` to APQ (automatic persisted queries)
  via the `@graphql-codegen/persisted-documents` preset.
